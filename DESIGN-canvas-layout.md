# Canvas Layout Architecture

> 캔버스 레이아웃 플러그인 시스템 설계 문서.
> freeform 네트워크 외에 타임라인, 캘린더 등 다양한 레이아웃을 지원하고,
> 이후 외부 플러그인으로 확장 가능한 구조를 목표로 한다.

---

## 1. 레이아웃 플러그인이란

하나의 캔버스가 노드를 **어떻게 배치하고, 어떻게 보여주고, 어떻게 조작하는지**를 통째로 정의하는 단위.

지금의 자유 배치 캔버스가 "freeform" 플러그인이고, 수평 타임라인이 또 하나의 플러그인이 된다. 향후 수직 타임라인, 캘린더 등도 같은 구조로 추가된다.

---

## 2. 플러그인의 네 가지 책임

### 배치 (Positioning)

노드를 캔버스 위 어디에 놓을 것인가.

- freeform: 사용자가 놓은 좌표를 그대로 사용한다. 자동 배치 없음.
- timeline: 노드의 시간 메타데이터 값에 따라 축 위의 위치를 자동 계산한다. 사용자가 저장한 좌표가 있으면 축과 직교하는 방향(레인)에만 반영한다.

### 시각 (Visual Layers)

캔버스의 배경과 오버레이에 무엇을 그릴 것인가.

**배경층** — 노드와 엣지 아래에 그려지는 레이어.
- freeform: 점 격자 (dot grid)
- timeline: 축 라인 + 시간 눈금(tick) + 눈금 라벨

**오버레이층** — 엣지 위, 노드 카드 아래에 그려지는 레이어.
- freeform: 없음
- timeline: 노드-축 커넥터 선(dashed), 축 위의 마커(dot), 기간 밴드(period band)

오버레이가 엣지 위 + 카드 아래인 이유: 커넥터 선이나 기간 밴드는 엣지 위에 보여야 하지만, 노드 카드를 가리면 안 된다.

### 인터랙션 (Interaction)

팬, 드래그, 줌 동작이 레이아웃마다 어떻게 달라지는가.

**팬 축 제한** — freeform은 자유. 특정 레이아웃은 한 축으로만 팬 가능.

**노드 드래그 축 제한** — freeform은 자유. 타임라인에서는 시간축 방향 드래그가 제한되거나, 드래그 결과를 시간값으로 역산해야 한다.

**추가 드래그 모드** — 타임라인의 기간 밴드 양쪽 끝을 잡아 늘리는 span resize처럼, 레이아웃 고유의 드래그 모드가 존재할 수 있다.

공통 인터랙션(노드 선택, 복수 선택, 컨텍스트 메뉴, 엣지 생성, Delete 키, Ctrl+A)은 레이아웃과 무관하게 동일하다.

### 노드 분류 (Node Classification)

같은 노드라도 레이아웃에 따라 완전히 다르게 그려질 수 있다.

- freeform: 모든 노드가 NodeCard로 그려진다.
- timeline: "사건" 노드는 NodeCard로 그려지지만, "시대" 같은 기간 노드는 카드가 아니라 축 위의 반투명 밴드로 그려진다.

따라서 플러그인은 "이 노드는 카드로 그려라(→ NodeLayer), 이 노드는 내가 오버레이에서 직접 그린다(→ OverlayComponent)"를 분류하는 책임을 가진다.

## 3. 데이터 흐름

### 도메인 독립적 메타데이터 매핑

MoC는 도메인 독립적 앱이다. 타임라인이 쓰는 "시간값"은 표준 날짜가 아니라 사용자가 정의한 숫자일 수 있다. 소설의 챕터 번호, 세계관 고유의 연호, 지질학의 Ma 단위 등.

이를 위해 세 계층으로 데이터가 흐른다:

**1단계: 사용자가 Archetype에 도메인 필드를 정의한다.**
"시대" archetype에 start_year, end_year 필드를 만들든, "소설 챕터" archetype에 chapter_number를 만들든 자유.

**2단계: Concept 인스턴스가 실제 값을 가진다.**
concept_properties 테이블에 각 concept × field 조합의 값이 저장된다. "조선시대"의 start_year = 1392, end_year = 1897.

**3단계: 캔버스 설정에서 사용자가 필드 매핑을 한다.**
"이 타임라인 캔버스에서 time_value로 쓸 필드가 뭐야?" — archetype별로 지정.

ConceptWorkspace가 중간에서 **번역**을 담당한다. concept_properties에서 실제 값을 읽고, 매핑 설정을 보고, 플러그인이 이해하는 표준 메타데이터(`time_value`, `end_time_value`, `role` 등)로 변환한다.

플러그인은 원본 도메인 필드를 전혀 모른다. "이 노드의 time_value는 1392"만 받는다. 이 번역 레이어가 도메인 독립성의 핵심이다.

### 플러그인의 데이터 요구 선언

각 플러그인은 자신이 필요로 하는 메타데이터 필드 목록을 선언한다.

- timeline: "time_value(필수, 숫자), end_time_value(선택, 숫자), role(필수, period 또는 occurrence)"
- calendar: "date(필수, 문자열)"
- freeform: 요구 필드 없음

이 선언을 기반으로 캔버스 설정 UI가 매핑 폼을 자동 생성한다.

## 4. 소유 관계

### Canvas가 layout을 직접 소유한다

Canvas 테이블에 두 컬럼이 추가된다:
- **layout**: 플러그인 키 (freeform, horizontal-timeline 등)
- **layout_config**: 해당 플러그인의 설정값 (JSON)

CanvasType은 분류용 메타데이터(이름, 아이콘, 색상)로 유지. 렌더링 동작을 결정하지 않는다.

### layout_config의 구조

두 종류의 설정이 들어간다:

**필드 매핑** — archetype별로 "내 도메인 필드 → 플러그인 요구 필드" 연결.
동일 캔버스에 여러 archetype의 노드가 공존할 수 있으므로, 매핑은 archetype별로 독립적이다.

**표시 설정** — unit(단위 접미사), tick 간격, 축 방향 등 플러그인 고유의 시각적 옵션. 어떤 설정이 가능한지는 플러그인이 configSchema로 선언한다.

---

## 5. 렌더링 층 순서

아래부터 위로:

1. **배경 (Background)** — 플러그인 제공. dot grid 또는 축+눈금.
2. **엣지 (EdgeLayer)** — 공통. 레이아웃과 무관하게 동일한 SVG 엣지.
3. **오버레이 (Overlay)** — 플러그인 제공 (선택). 커넥터, 밴드, 마커 등.
4. **노드 카드 (NodeLayer)** — 공통. 플러그인의 분류 결과에서 "카드" 노드만 렌더링.
5. **UI (Controls, Menus, Modals)** — 공통.

---

## 6. 노드 드롭 후 처리

freeform에서는 드래그가 끝나면 새 좌표를 position_x/y에 저장하면 된다.

타임라인에서는 추가 처리가 필요하다. 노드를 드래그해서 놓으면 새 좌표를 시간값으로 역산하여 concept_properties를 업데이트해야 한다. 예: "X가 200px 이동 → time_value가 1 증가".

이 역산 로직은 플러그인마다 완전히 다르므로, 플러그인이 "드롭 후 처리"를 정의하고 ConceptWorkspace가 그 결과(위치 저장 + property 업데이트)를 실행하는 구조.

---

## 7. 플러그인 등록과 조회

앱 시작 시 빌트인 플러그인들을 등록한다. 캔버스를 열 때 canvas.layout 값으로 해당 플러그인을 조회한다. 없으면 freeform으로 fallback.

등록 목록은 캔버스 설정 UI의 레이아웃 선택 드롭다운에도 사용된다.

---

## 8. 외부 플러그인 전환 경로

현재는 앱 내부에 직접 구현하되, 플러그인 인터페이스를 순수 객체(컴포넌트 + 함수)로 유지한다.

향후 외부 플러그인 지원 시:
- 플러그인 = npm 패키지 또는 로컬 JS 파일
- 앱 시작 시 플러그인 디렉토리를 스캔 → dynamic import → 동일한 인터페이스로 등록
- 내부 구조 변경 없음. 등록 경로만 추가.

| freeform | 현재 동작을 플러그인으로 래핑 | Phase 1 |

| vertical-timeline | 수직 시간축 타임라인 | Phase 2 |
| calendar | 달력 격자 배치 | Phase 3 |
